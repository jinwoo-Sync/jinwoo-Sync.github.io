인도네시아 프로젝트에서 항공 측량 데이터를 처리하는 GIS 파이프라인을 구축하고, Python → C# 컨버팅을 통해 처리 속도를 33배 개선한 과정을 정리한다.

## 배경

인도네시아 항공 측량 데이터는 LAS(포인트클라우드), TIF(래스터), SHP(벡터) 등 다양한 GIS 포맷으로 구성되어 있다. 연구팀에서 Python으로 개별 처리 스크립트를 개발했지만, 분할 개발된 코드들이 산재해 있었고 처리 성능에 한계가 있었다.

## 파이프라인 구조

전체 데이터 처리 시퀀스를 3단계로 정리했다:

### 1단계: SHP 파일 생성

```
TIF → PNG 변환 → Airborne + Global Building + OSM 결합 → .shp
```

TIF 이미지를 PNG로 변환한 뒤, 항공 데이터와 글로벌 건물 데이터, OpenStreetMap 데이터를 결합하여 SHP(Shapefile)를 생성한다.

이 과정에서 겹치는 영역이 발생하는데, 중복 영역을 자동으로 합치는 기능을 구현하여 수작업을 제거했다.

### 2단계: LAS 핸들링

```
포인트클라우드 추출 → Polygon 분할 + Multi-layer 분할 → .las
```

SHP 결과물이 나오면 2단계와 3단계를 동시에 진행할 수 있다.

### 3단계: OBJ 메쉬 생성

```
LOD1 Mesh / LOD2 Mesh → .obj
```

## DTM 생성 최적화

DTM(Digital Terrain Model) 생성이 전체 파이프라인에서 가장 시간이 많이 소요되는 부분이었다.

### 기존 방식: PDAL CSF

기존에는 PDAL 명령어를 이용한 CSF(Cloth Simulation Filter) 방식으로 DTM을 생성했다. 이 방식은 69GB 데이터 기준 약 36시간이 소요됐다.

### 개선된 DTM 생성 알고리즘

C#으로 컨버팅하면서 DTM 생성 알고리즘을 다음 5단계로 재설계했다:

**Step 1 — Voxel 기반 지면 점 추출**

전체 점군을 **5m × 5m 격자(Voxel)**로 나눈다. 각 격자 안의 점들을 Z값(높이) 기준으로 정렬한 뒤, Z 범위가 1m 미만이면 전부 지면으로 판단한다. 그 이상이면 **최솟값 + 2m(또는 전체 범위의 15%)** 이하인 점들만 지면으로 선택한다. 즉 "가장 낮은 클러스터"만 남기고 위에 있는 건물·나무는 제거한다.

**Step 2 — 건물 지붕 추가 제거**

이번엔 **25m × 25m 블록** 단위로 분석한다. 각 블록의 점들로 최소제곱법 평면 피팅을 수행하여 법선 벡터의 Z 성분을 계산한다. 법선이 거의 수직(Z ≈ 1.0, 즉 수평 평면)이고 주변 블록과 높이 차가 3.5m 이상이면 건물 지붕으로 판단하고 제거한다.

**Step 3 — MinZ 그리드 생성**

지면 점들을 픽셀 그리드에 매핑하여, 각 셀에서 **가장 낮은 Z값**만 기록한다. 점이 없는 셀은 NoData(-9999)로 표시한다.

**Step 4 — 구멍 메우기**

NoData 셀을 주변 유효 셀의 값으로 보간하여 채운다.

**Step 5 — 스무딩**

**1회만** 평균 스무딩을 적용한다. 논밭, 강 등 지형 특징 보존을 위해 최소한으로만 수행하는 것이 핵심이다.

## 성능 결과

| 지표 | 기존 (PDAL CSF) | 개선 후 (C# 파이프라인) |
|------|-----------------|----------------------|
| 입력 데이터 | 69 GB (원본) | 69 GB (원본) |
| 처리 시간 | 2,160분 (36시간) | 약 60분 (1시간) |
| 처리 속도 | 32.71 MB/min | 약 1,177 MB/min |
| 개선 배율 | - | **약 36배 개선** |

**Trade-off 활용**: 정밀도를 일부 타협하는 대신 속도를 극대화하여, 실 사용자들이 외부 DTM 데이터 없이도 프로그램 내부에서 즉시 작업을 수행할 수 있도록 최적화했다.

## Python → C# 컨버팅에서 배운 점

1. **시퀀스 정리가 먼저**: 분할 개발된 코드의 동작 순서를 먼저 정리하고 통일한 뒤 컨버팅을 진행해야 한다
2. **Python 가상환경 임베딩**: C# 프로그램에서 Python 스크립트를 직접 호출하는 구조를 만들면, 일부 라이브러리(PDAL, GDAL 등)는 Python 네이티브를 유지하면서도 전체 시스템은 C#으로 통합할 수 있다
3. **GIS 데이터의 특성**: LAS, TIF, SHP 등 GIS 데이터는 포맷별 특성을 이해해야 효율적인 파이프라인을 설계할 수 있다

## 기술 스택

- Python, C#, PDAL, GDAL/OGR
- SAM (Segment Anything Model), EfficientNet
- LAS, TIF, SHP, OBJ, PNG
