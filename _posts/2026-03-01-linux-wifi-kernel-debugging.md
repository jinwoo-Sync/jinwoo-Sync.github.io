H그룹 용역 프로젝트에서 Intel NUC의 WiFi가 야외 테스트 중 반복적으로 다운되는 문제를 겪었다. 이 글은 문제를 추적하고 해결하기까지의 19단계 디버깅 과정을 기록한 글이다.

## 환경

- **하드웨어**: Intel NUC + Intel BE201 WiFi 칩
- **OS**: Ubuntu 24.04
- **용도**: NUC Hotspot을 통한 현장 장비 원격 제어

## 문제 발생

초기 세팅에서 NetworkManager를 이용해 NUC Hotspot을 구성했다. 초기에는 한 번의 NetworkManager reset으로 안정적으로 동작했고, 야외 테스트/캘리브레이션 데이터 취득 시에도 큰 이상은 없었다.

그런데 실제 야외 장기 테스트에서 **WiFi가 예고 없이 다운되는 장비들**이 발견되기 시작했다.

## 디버깅 과정

### 1단계: NetworkManager → hostapd/dnsmasq 전환

NUC Hotspot이 꺼지는 이슈를 확인하고, NetworkManager 대신 `hostapd`와 `dnsmasq`를 직접 관리하는 방식으로 전환했다.

### 2단계: 네트워크 설정 강화

안정성 확보를 위해 추가 Config 설정을 직접 작성:
- Channel, hw_mode 직접 지정
- 호환성 옵션 추가
- 보안 설정 강화

### 3단계: 로그 분석 — 에러 체인 확인

`dmesg` 로그를 분석한 결과, 다음과 같은 에러 체인이 반복적으로 발생하고 있었다:

```
wireless API 에러 → 펌웨어 crash → reprobe → PCIe reset → USB 실패 → SSD 다운
```

WiFi 문제가 SSD까지 영향을 미치고 있었고, 이로 인해 로깅 툴에서 에러 팝업이 발생하는 상황이었다.

### 4단계: hostapd 직접 설정

NetworkManager가 아닌 `hostapd`, `dnsmasq` 설정 파일을 직접 생성하고 네트워크 인터페이스를 직접 핸들링했다.

어느 정도 해결된 것처럼 보였으나...

### 5단계: 양재 데이터 취득에서 재발

양재 현장 데이터 취득 시에 다시 Hotspot down 및 SSD 누락이 발견됐다.

핵심 에러 메시지:

```
Failed to send LINK_CONFIG_CMD (-5)
```

이 플래그는 **드라이버 내부에서 펌웨어가 명령을 거부**했다는 의미이다.

### 6단계: 커널-하드웨어 매칭 의심

이 시점에서 하드웨어(BE201 칩)와 커널의 매칭 문제를 의심하기 시작했다. [Arch Linux 포럼](https://bbs.archlinux.org/viewtopic.php?id=303611)에서 유사한 사례를 확인했다.

### 7단계: 신규 펌웨어 업데이트

최신 펌웨어를 다운로드하여 업데이트했다. 하지만 **커널에서는 여전히 구버전 펌웨어를 로드**하는 문제가 발생했다.

### 8단계: 펌웨어 파일 강제 교체

커널이 ucode 98 대신 96을 찾고 있어서, `98.ucode` 파일을 `96.ucode` 이름으로 바꿔치기하고 심볼릭 링크로 연결을 시도했다.

### 9단계: Backports 빌드

리눅스 `backport-iwlwifi` git repo를 직접 clone하여 빌드:
- `flex`, `bison` 라이브러리 사전 설치
- 소스 직접 빌드 후 모듈 로드

### 10~11단계: 빌드 후 테스트

빌드 로그에서는 실패로 표시됐지만 실제로는 모듈이 로드된 것으로 보여 테스트를 진행했다. **ucode 96 버전이 로드된 것을 확인**했지만, 여전히 동일한 에러가 발생했다:

```
iwlwifi 0000:00:14.3: Failed to run INIT ucode: -5
```

### 12~13단계: 하드웨어 문제 확신

이 시점에서 Intel WiFi HW 모듈과 신규 NUC의 호환성 문제라고 확신했다. 그럼에도 이전 설정들을 다시 시도했지만 `NMI_INTERRUPT_UMAC_FATAL` 에러가 계속 발생했다.

### 14단계: 커널 다운그레이드

Linux 커널을 다운그레이드하여 재시도했지만 동일한 결과.

### 15단계: 커널 공식 버그 확인

리눅스 커널 공식 버그 트래커에서 **Bug #220034**로 등재된 이슈임을 확인했다. 해결 버전인 linux-firmware 6.14.11을 설치했지만 여전히 동작하지 않았다.

### 16~17단계: 팀 보고 및 OS 전환

하드웨어 문제라고 판단하고 팀장님께 보고했다. 설 연휴 동안 팀장님이 **Ubuntu 20.04**로 OS를 낮춰서 진행하는 방향으로 전환했다.

### 18단계: WiFi 칩 교체

BE201보다 하위 버전의 WiFi 칩을 구매하여 장착하고 사내 테스트를 통과했다.

### 19단계: 현장 적용

그럼에도 등촌 현장 취득 시점에는 여전히 이슈가 남아있어 결국 키보드/마우스 직접 연결로 데이터 취득을 진행했다.

## 근본 원인

이 문제는 **Linux 커널 공식 버그(Bug #220034)**로 등재된 이슈이다.

- **버그 링크**: [bugzilla.kernel.org/show_bug.cgi?id=220034](https://bugzilla.kernel.org/show_bug.cgi?id=220034)
- **해결 예정 버전**: 커널 6.14.4 이상
- Intel 커널 개발자 Johannes Berg가 근본적인 해결(config-cleanup)이 완료되기 전까지 문제 코드를 되돌리는 패치를 적용하겠다고 답변

## 교훈

1. 최신 하드웨어 + 최신 Linux 커널 조합이 항상 안정적이지는 않다
2. `dmesg` 에러 체인을 끝까지 추적하면 근본 원인에 도달할 수 있다
3. 커널 버그 트래커를 확인하는 것이 디버깅 시간을 크게 줄여줄 수 있다
4. 현장 배포에서는 항상 폴백 방안(유선 연결 등)을 준비해야 한다
